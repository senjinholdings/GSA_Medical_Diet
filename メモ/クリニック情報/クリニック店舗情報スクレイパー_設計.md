# クリニック店舗情報スクレイパー 設計書

最終更新: 2025-08-12  
対象要件: `クリニック店舗情報スクレイパー_要件定義.md`

---

## 1. 概要
- クリニック公式サイトのトップURLを起点に同一ドメイン配下をクロールし、院（店舗）情報を抽出。
- 出力は既存CSV `public/mouthpiece/data copy/出しわけSS - stores.csv` へ最下部に追記（append）。

## 2. スコープ確認（要点）
- ドメイン内クロール、robots.txt順守、サイトマップ取り込み、限定的JSレンダリング。
- 必要CSV列: `store_id, clinic_name, store_name, Zipcode, adress, access`

## 3. アーキテクチャ概要
- 構成要素:
  - Fetcher: httpxでHTTP取得（キャッシュ/再試行/robots対応）。
  - Renderer: Playwright（必要URLのみ）。
  - URL Frontier: 訪問予定/済管理、最大深さ/件数制御、除外パターン。
  - Parser: HTML→DOM、`extruct`でJSON-LD/Microdata抽出。
  - PageClassifier: 一覧/詳細/その他の判定（ヒューリスティクス+キーワード）。
  - Extractor: 詳細ページから院情報を抽出（構造化データ優先→DOM）。
  - Normalizer: 郵便番号・住所・表記整形、アクセステキスト生成。
  - RecordBuilder: CSV行スキーマへマッピング。
  - Deduper: （任意）既存CSVとの重複判定（`clinic_name+store_name`など）。
  - CsvAppender: 指定CSVに追記（排他制御、ヘッダ検証）。
  - Logger: JSONログ、統計。
  - CLI: Typerベース、YAML設定読込。

## 4. データフロー
1) 入力URLをFrontierへ投入→robots/sitemapで初期URL拡張  
2) Fetcherで取得（必要に応じRenderer）  
3) Parser→PageClassifier  
4) 一覧: 子リンク抽出しFrontierへ投入／詳細: Extractor→Normalizer  
5) RecordBuilder→CsvAppender（即時追記）  
6) 継続 until 上限到達 or Frontier空

## 5. 主要データ構造（内部）
- UrlItem: `{ url, depth, referrer }`
- PageData: `{ url, status, html, final_url, rendered:boolean }`
- ExtractedClinic: `{ clinicBrandName, branchName, postalCode, addressFull, nearestAccess, sourceUrl, raw }`
- CsvRow: `[store_id, clinic_name, store_name, Zipcode, adress, access]`

## 6. CSVマッピング仕様
- `store_id`: 〈ブランドキー〉_〈3桁連番〉
  - ブランドキー: 既存CSV内の同`clinic_name`で使用中の`store_id`接頭辞を優先再利用（例: `urara`, `sbc`）。
  - 新規ブランド時: ドメイン名/和名のローマナイズから短いスラグ（例: "Oh My Teeth"→`omt`、重複時は`omt2`）。
  - 連番: 既存CSV内で当該ブランドキー最大番号+1（ゼロパディング3桁: `001`）。
- `clinic_name`: 抽出ブランド名。構造化データ `name`/`legalName`→ヘッダ/フッタ表記→手掛かり（サイトタイトル）で決定。
- `store_name`: 院名（例: 「銀座院」「新宿院」）。`
- `Zipcode`: `〒\d{3}-\d{4}`形式。`〒`がない場合は付与。数字7桁のみ検出時は`XXX-XXXX`へ整形。
- `adress`: 都道府県〜建物名までのフル住所（全角・半角/スペース正規化を実施）。
- `access`: 最寄駅+徒歩分の短文（例: 「銀座駅から徒歩5分」）。無ければ`-`。

バリデーション: ヘッダ一致確認、カンマ/改行のクオート、`store_id`空はスキップ。

## 7. クロール戦略
- 範囲: 同一ドメイン（設定でサブドメイン可）。
- 除外: `/wp-admin`, `/assets`, `/static`, `/privacy`, `/terms`, 拡張子 `.pdf,.jpg,.png,.svg,.webp,.css,.js` 等。
- 初期探索: トップ+`/sitemap.xml` → クリニック一覧候補URLを優先（URL/文言ヒューリスティクス）。
- 深さ/件数: 既定 `depth<=3`, `pages<=2000`。
- レンダリング: 一覧/詳細でDOMが未充足のときのみPlaywright。

## 8. ページ判定・抽出ロジック
優先度: JSON-LD > Microdata/RDFa > OGP > DOMヒューリスティクス
- 一覧検出: 「院一覧/医院一覧/クリニック一覧/店舗一覧/拠点」等のテキスト or `a[href*='/clinic/']`多数。
- 詳細検出: 住所/電話/診療時間/アクセスの同居、JSON-LD `LocalBusiness`/`MedicalClinic`あり等。
- 抽出要素:
  - ブランド名: サイト共通ヘッダ/フッタ、構造化データ `organization.name`。
  - 院名: `h1`, `.clinic-name`, パンくず末尾。
  - 住所: `address`, `#address`, ラベル対（「住所」「所在地」）。
  - 郵便番号: 住所テキストから正規表現抽出。
  - アクセス: 「アクセス」「最寄駅」周辺テキスト、表の見出し。

## 9. 正規化ルール（抜粋）
- 郵便番号: 全角→半角、`NNN-NNNN`、先頭に`〒`付与。
- 住所: 連続スペース縮約、全角英数→半角、ビル名はそのまま保持。
- アクセス: 「徒歩」「分」パターンを優先抽出。複数候補時は最短徒歩時間を採用。

## 10. 重複判定（任意機能）
- 既存CSV内の重複: `clinic_name + store_name` 完全一致でスキップ（`--dedupe-existing`時）。
- セッション内重複: `url`正規化、`clinic_name + postalCode + branchName`のハッシュでユニーク化。

## 11. 失敗時動作
- HTTP: 429/5xxは指数バックオフで最大3回再試行。
- 解析フォールバック: 構造化データ→DOM→レンダリング。
- CSV書き込み: ファイルロック（OSロックかポータブルロックファイル）で排他。失敗時は再試行1回。

## 12. 設定ファイル（YAML）
```yaml
start_url: https://example-clinic.jp/
allow_subdomains: false
max_pages: 2000
max_depth: 3
concurrency: 3
delay_seconds: 0.5
rendering:
  enabled: true
  url_allow_patterns: ["/clinic/", "院一覧", "医院一覧", "アクセス"]
  wait_for_selectors: [".clinic-detail", "#address"]
extractors:
  domain_overrides:
    "example-clinic.jp":
      list_selectors: [".branch-list a", "a[href*='/clinic/']"]
      detail_selectors:
        name: ["h1", ".clinic-name"]
        address: [".address", "#address", "address"]
        access: [".access", "#access"]
output:
  csv_append_path: "/Users/hattaryoga/Desktop/kiro_2_サイト構成分析/public/mouthpiece/data copy/出しわけSS - stores.csv"
  save_html_snapshot: false
  dedupe_existing: false
```

## 13. CLI設計
- ライブラリ: Typer  
- コマンド例:
```bash
clinic-scrape \
  --url https://example-clinic.jp/ \
  --config config.yaml \
  --csv-append-path \
  "/Users/hattaryoga/Desktop/kiro_2_サイト構成分析/public/mouthpiece/data copy/出しわけSS - stores.csv" \
  --max-pages 2000 \
  --max-depth 3 \
  --concurrency 3 \
  --rendering auto
```
- 主要オプション:
  - `--url`: 起点URL（必須）
  - `--csv-append-path`: 出力CSVパス（必須）
  - `--max-pages`, `--max-depth`, `--concurrency`, `--delay-seconds`
  - `--rendering [on|off|auto]`
  - `--dedupe-existing`

## 15. 擬似コード（要点）
```python
# cli.py
cfg = load_config()
validate_csv_header(cfg.csv_append_path,
  ["store_id","clinic_name","store_name","Zipcode","adress","access"])
frontier = Frontier(start_url=cfg.start_url)
while frontier.has_next() and limits_not_reached():
    url_item = frontier.next()
    page = fetch_or_render(url_item, cfg)
    page_type = classify_page(page)
    if page_type == "list":
        links = extract_links(page)
        frontier.add_many(links)
    elif page_type == "detail":
        data = extract_clinic(page)
        norm = normalize(data)
        row = to_csv_row(norm, csv_path=cfg.csv_append_path)
        if not (cfg.dedupe_existing and exists_in_csv(row)):
            append_csv_row(cfg.csv_append_path, row)
```

## 17. テスト計画
- 単体: 正規化（郵便/住所/アクセス）、ヘッダ検証、`store_id`生成規則、URL正規化。
- 疑似統合: extructのJSON-LDサンプル、DOMスニペットからの抽出に対する期待CSV行。
- 結合: VCR.pyでHTTP録画、同一結果の再現性。
- ゴールデン: 既存CSVに対し、指定ドメインを1回走らせた差分が期待値と一致。

## 18. ログ/監視（最小）
- レコード追加時に `INFO` ログ: `{store_id, clinic_name, store_name, url}`
- スキップ/重複/エラーは `WARN/ERROR` で詳細出力。

## 19. リスクと回避
- JS依存強→Playwrightの利用をURL条件で許可。
- 強いWAF→レート制御/遅延を設定可能に。
- CSV壊れリスク→追記前バックアップ（任意）、ロールバック時は末尾N行を削除できるユーティリティ提供。

## 20. 変更容易性
- ドメイン固有セレクタはYAMLで上書き可。
- 新規列追加は`RecordBuilder`とヘッダ検証の更新で対応。
